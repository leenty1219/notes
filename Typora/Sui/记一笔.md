这套“记一笔”页面，本质上是 5 层协作：

1. Container/Generator 决定“要生成哪个页面、共用哪份 sharedTransaction”
   入口在 [RecordTransContainerBuilder.swift (line 62)](app://-/index.html?hostId=local#) 和 [RecordPageFactory.swift (line 187)](app://-/index.html?hostId=local#)。
   builder 先创建一份共享的 RecordSharedTransaction，再按 tradeType 生成各 page generator。generator 真正 generate() 时才创建 context + table + pageVC，并且默认缓存页面实例，见 [RecordPageGenerator.swift (line 16)](app://-/index.html?hostId=local#)。
2. Context 决定“这一页有哪些 managers / rows，以及页面生命周期怎么同步业务状态”
   基类是 [RecordTransactionPageContext.swift (line 17)](app://-/index.html?hostId=local#)。
   rows 不是自己存的，而是实时从 managers -> unfoldedManagers -> row 算出来，见同文件 [62-64 (line 62)](app://-/index.html?hostId=local#)。
   所以“页面展示哪些 cell”，本质由 managers 和每个 manager 的 isFold 决定。
3. RowManager 决定“某个业务字段如何映射到 row/publisher/transaction”
   抽象在 [RecordRowManager.swift (line 19)](app://-/index.html?hostId=local#)。
   它有三类职责：
   update(...) 从 transaction/sharedTransaction 灌状态；
   config(...) 把当前 UI 状态写回 transaction；
   row 提供真正的展示对象。
4. Row 决定“cell 长什么样、何时 reload、如何进入编辑态”
   基类在 [RecordBaseRow.swift (line 50)](app://-/index.html?hostId=local#) 和 [RecordInputRow.swift (line 24)](app://-/index.html?hostId=local#)。
   Row 持有 RecordPublisher，publisher 变化时通常触发 row.reload()，见 [RecordPublisher.swift (line 30)](app://-/index.html?hostId=local#)。
5. Cell 决定“first responder / inputView / UI 布局”
   基础输入 cell 在 [RecordInputCell.swift (line 15)](app://-/index.html?hostId=local#)。
   它在 becomeFirstResponder() 时向 row 取 inputView/inputAccessoryView，并把 presentation.isSelected 改成 true，见 [165-189 (line 165)](app://-/index.html?hostId=local#)。

**cell 加载 / 出现链路**

从 [RecordPageViewController.swift (line 16)](app://-/index.html?hostId=local#) 切入，首次出现顺序是：

1. RecordPageFactory.transPageGenerator 创建 context，如果有 editTransaction 先 context.new(transaction)，然后立刻用 context.rows 创建初始 section，见 [RecordPageFactory.swift (line 188)](app://-/index.html?hostId=local#)。
2. RecordPageViewController.init 时把 context.delegate = self，并创建 ChangeableManager，见 [RecordPageViewController.swift (line 27)](app://-/index.html?hostId=local#)。
3. viewDidLoad 做 4 件关键事：
   context.pageDidLoad()；
   绑定 tableView.delegate/dataSource = table；
   table.delegate = self，也就是 pageVC 自己接收 SMUITable 的 diff 回调；
   setDelegateForRows(delegate: self)，把所有实现了 RecordChangeable 的 row 统一挂到 ChangeableManager，见 [46-49 (line 46)](app://-/index.html?hostId=local#) 和 [197-205 (line 197)](app://-/index.html?hostId=local#)。
4. viewWillAppear 后，真正重要的是 context.pageWillAppear()。
   对交易页，这会继续走到 RecordTagPageContext.pageWillAppear 和 RecordAmountPageContext.pageWillAppear：
   金额页先把 sharedTransaction -> amountRowManager，见 [RecordAmountPageContext.swift (line 22)](app://-/index.html?hostId=local#)；
   标签页再把 sharedTransaction -> date/memo，并给当前展开的 editor 做 reload，见 [RecordTagPageContext.swift (line 28)](app://-/index.html?hostId=local#)。
5. viewDidAppear 后，context.didAppear = true，然后 context.pageDidAppear()。
   金额页如果 sharedTransaction.inputingType == .amount，会自动恢复金额输入，见 [RecordAmountPageContext.swift (line 32)](app://-/index.html?hostId=local#)。
   标签页如果 editor 标记了 needsRestoreEditing，会让 pageVC 调 changeableManager.restoreLastChangeable()，见 [RecordTagPageContext.swift (line 41)](app://-/index.html?hostId=local#) 和 [RecordPageViewController.swift (line 221)](app://-/index.html?hostId=local#)。

这里最关键的一点是：
tableView 并不主动“拥有业务状态”，它只是把 context.rows 当前算出来的结果显示出来。
所以你以后加退款 cell，核心不是往 VC 里塞 UI，而是决定它应该由哪个 Context.managers 输出、什么时候 fold/unfold、以及和 sharedTransaction 怎样双向同步。

**cell 变动 / 刷新链路**

这里有三条并行链路，最容易混。

1. “字段值变了，但 rows 集合没变”
   典型如金额、备注内容变化。
   publisher.setValue(...) -> row observer -> row.reload()，见 [RecordAmountRow.swift (line 57)](app://-/index.html?hostId=local#) 和 [RecordMemoRow.swift (line 23)](app://-/index.html?hostId=local#)。
   row.reload() 最后会走到 changeableDidChange(needsReload: false)，page 层的 ChangeableManager.changeableDidChangeContent 会先让 table.reloadData() 更新数据模型，但临时屏蔽 table delegate，避免触发整表 diff UI，见 [RecordPageViewController+Changeable.swift (line 70)](app://-/index.html?hostId=local#) 和 [107-120 (line 107)](app://-/index.html?hostId=local#)。
   这个设计的目的，是“数据结构更新了，但 UI 尽量局部自己改，不走整表动画”。
2. “row 高度变了”
   典型是备注。
   RecordMemoRow.reload() 里会 updateHeight()，高度变了就 changeableDidLayout()，见 [RecordMemoRow.swift (line 62)](app://-/index.html?hostId=local#)。
   page 层收到后只是 tableView.beginUpdates/endUpdates 让系统重算高度，再确保当前编辑控件可见，见 [RecordPageViewController+Changeable.swift (line 93)](app://-/index.html?hostId=local#)。
   所以备注是“自持 cell + 自持高度”的特殊行。
3. “rows 集合变了”
   典型是折叠/展开、tradeType 切换、批量 preset/edit。
   这时由 Context 主动调 delegate?.context(self, didReloadRows: rows, animated: ...)，pageVC 收到后直接替换 table.sections 并 table.reloadData()，见 [RecordPageViewController.swift (line 208)](app://-/index.html?hostId=local#)。
   如果 animated = true，SMUITable 先算 diff，再由 indexPathResultDidChange 把 insert/delete/move 应到 UITableView，见 [168-191 (line 168)](app://-/index.html?hostId=local#)。
   如果 animated = false，就是整表 reloadData()。

batchUpdate 是专门为第 3 条准备的。
Context.batchUpdate 会先把 delegate 置空，执行多个 manager 更新，最后一次性 didReloadRows，避免中间多次 reload 冲突，见 [RecordTransactionPageContext.swift (line 295)](app://-/index.html?hostId=local#)。

**rowManager、row、inputSource、editor 怎么协同**

以分类为例最清楚：

1. RecordCategoryRowManager 内部并不直接查接口，而是持有 RecordDataPublisher.category(...) 生成的两个 publisher：
   presentation 给 cell 显示标题/详情；
   inputSource 给 picker/drawer 显示数据源，见 [RecordCategoryRowManager.swift (line 33)](app://-/index.html?hostId=local#)。
2. RecordDataPublisher 监听 dataManager 的最近使用和全量数据；两者都到了以后，组装成 RecordTableInputSource，并尽量恢复之前的选中项，见 [RecordDataPublisher.swift (line 33)](app://-/index.html?hostId=local#) 和 [58-67 (line 58)](app://-/index.html?hostId=local#)。
3. RecordTagRowManager.updateSelection(with:) 负责“编辑已有 transaction 时，根据 id 找回 inputSource 中的选中项”。如果数据还没拉完，就先缓存 selectedTagIDToBeUpdatedWhenDataAvailable，等 inputSource 有值后再补上，见 [RecordTagRowManager.swift (line 34)](app://-/index.html?hostId=local#)。
4. RecordCategoryRow 持有 inputSource，同时自己决定使用哪种 inputView。
   它可能返回 drawer / picker / 普通分类输入 view，但都共享同一个 inputSource，见 [RecordCategoryRow.swift (line 45)](app://-/index.html?hostId=local#)。
5. RecordInputCell.becomeFirstResponder() 时会向 row 取 inputView，所以“点 cell 弹出选择器”不是 VC 写死的，而是 row 自己提供输入视图，见 [RecordInputCell.swift (line 165)](app://-/index.html?hostId=local#)。
6. editor 是“增删改查基础数据”的外部协作层。
   RecordTagPageContext.pageWillAppear() 会把当前展开行的 editor 绑到自己身上，并在必要时 reloadIfNeeded()，见 [RecordTagPageContext.swift (line 33)](app://-/index.html?hostId=local#)。
   所以 editor 不直接驱动 table，它只负责在基础数据变更后触发 dataManager 重新拉数，最终还是回流到 inputSource/presentation -> row.reload() 这条链。

**折叠 / 胶囊 / 编辑恢复**

这块对你加退款来源 cell 很关键。

1. 折叠能力不是 row 才有，是 manager 先有。RecordRowEntry 上就定义了 isFold/isFoldable/fold/unfold，见 [RecordRowManager.swift (line 19)](app://-/index.html?hostId=local#)。
2. RecordTagPageContext.capsulableManagers 统一给 date/member/corporation/project 配 fold/unfold 闭包。
   fold 时：改 isFold = true，整页重算 rows；
   unfold 时：改 isFold = false，整页重算 rows，然后下一帧自动 row.select() 进入编辑，见 [RecordTagPageContext.swift (line 103)](app://-/index.html?hostId=local#)。
3. capsuleRow 不是静态文案，它监听每个 entry 的 isFold，把“已折叠的 manager”转成胶囊项，见 [RecordCapsuleRow.swift (line 23)](app://-/index.html?hostId=local#)。
   所以某个字段要支持“收成胶囊”，前提不是做一个 capsule cell，而是让它对应的 manager 成为 capsulableManagers 的一员并实现 entryTitle。
